Findings:

1. `project_symbols` can overwrite an explicit `symbols` result of `0`, which breaks the Track E0-2 precedence rule. In [src/impact/hotspots.rs](C:/dev/ChangeGuard/src/impact/hotspots.rs:183) the fallback gap set is defined as “missing from primary OR complexity is 0”, and the fallback then overwrites that entry at [src/impact/hotspots.rs](C:/dev/ChangeGuard/src/impact/hotspots.rs:227). Track E0-2 says fallback should fill only files absent from `symbols`, with `symbols` always winning on overlap, even when both tables have the same file ([conductor/trackE0-2/spec.md](C:/dev/ChangeGuard/conductor/trackE0-2/spec.md:39), [conductor/trackE0-2/spec.md](C:/dev/ChangeGuard/conductor/trackE0-2/spec.md:114)). This can inflate hotspot complexity for files whose impact-run complexity is intentionally zero.

2. The fallback complexity formula does not match the spec or the primary query, so hotspot scores are not comparable across the two sources. The primary path uses `MAX(IFNULL(cognitive_complexity, 0), IFNULL(cyclomatic_complexity, 0))` at [src/impact/hotspots.rs](C:/dev/ChangeGuard/src/impact/hotspots.rs:162), matching the spec ([conductor/trackE0-2/spec.md](C:/dev/ChangeGuard/conductor/trackE0-2/spec.md:22), [conductor/trackE0-2/spec.md](C:/dev/ChangeGuard/conductor/trackE0-2/spec.md:34)). The fallback instead computes `AVG(cognitive_complexity)` and ignores `cyclomatic_complexity` at [src/impact/hotspots.rs](C:/dev/ChangeGuard/src/impact/hotspots.rs:206). That is a behavioral deviation, not just an implementation detail; it can under-rank complex files in the fallback path.

3. The fallback safety check is incomplete for table existence. The code guards only `project_symbols` via `table_exists("project_symbols")` at [src/impact/hotspots.rs](C:/dev/ChangeGuard/src/impact/hotspots.rs:195), but the actual fallback query also depends on `project_files` via an inner join at [src/impact/hotspots.rs](C:/dev/ChangeGuard/src/impact/hotspots.rs:208). If `project_symbols` exists but `project_files` is absent or mismatched, this path errors instead of degrading gracefully. That is weaker than the Track E0-2 “do not crash” requirement for the fallback path ([conductor/trackE0-2/spec.md](C:/dev/ChangeGuard/conductor/trackE0-2/spec.md:51)).

4. `save_packet` is not batched in a transaction, so SQLite write amplification is higher than it needs to be. [src/state/storage.rs](C:/dev/ChangeGuard/src/state/storage.rs:195) inserts the snapshot, then [src/state/storage.rs](C:/dev/ChangeGuard/src/state/storage.rs:312) loops per changed file, then calls `persist_symbols`, all on a bare `Connection` with no surrounding transaction. For large scans this means repeated statement/commit overhead and weaker atomicity. Given your review focus on SQLite batching, this is the main performance concern in `storage.rs`.

5. `table_exists_and_has_data` is not fully table-name-safe. It correctly checks existence first, but then builds `SELECT EXISTS(SELECT 1 FROM {table_name} LIMIT 1)` with `format!` at [src/state/storage.rs](C:/dev/ChangeGuard/src/state/storage.rs:411). Because identifiers cannot be parameter-bound, this helper should either constrain names to a trusted set or quote/escape them centrally; as a public method it currently assumes trusted callers. That is a safety footgun in the table-existence utility itself.
