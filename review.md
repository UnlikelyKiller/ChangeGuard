**Findings**

- `Critical` The new `post-commit` path appears to roll back most non-trivial commits instead of promoting them. [hook_post_commit.rs](C:/dev/ChangeGuard/src/commands/hook_post_commit.rs:44) builds a `CommitRequest` with `verification_status: None` and then calls `commit_change(..., false)` at [hook_post_commit.rs](C:/dev/ChangeGuard/src/commands/hook_post_commit.rs:55). But `commit_change` rejects `Architecture`/`Feature`/`Bugfix`/`Infra` transactions without verification when `verify_to_commit` is enabled at [transaction.rs](C:/dev/ChangeGuard/src/ledger/transaction.rs:145). Since category inference maps common prefixes like `feat`, `fix`, `build`, `revert` into those categories, successful Git commits can still lose their pending ledger record in `post-commit`.

- `High` The sidecar hash is written but never enforced, so an aborted commit can be incorrectly promoted by a later unrelated commit. The sidecar stores `commit_msg_hash` at [hook_commit_msg.rs](C:/dev/ChangeGuard/src/commands/hook_commit_msg.rs:359), but `post-commit` only leaves a comment saying verification is optional and proceeds anyway at [hook_post_commit.rs](C:/dev/ChangeGuard/src/commands/hook_post_commit.rs:32). That means a stale `.changeguard/state/pending_hook_tx` can survive an aborted commit and be committed on the next `post-commit`, which reintroduces the original “wrong commit/provenance” class of bug.

- `High` Hook-created signatures are almost certainly invalid against the persisted ledger entry. The signature is generated in `commit-msg` using a pre-commit timestamp at [hook_commit_msg.rs](C:/dev/ChangeGuard/src/commands/hook_commit_msg.rs:323), stored in the sidecar, and then reused verbatim by `commit_change` when `req.signature.is_some()` at [transaction.rs](C:/dev/ChangeGuard/src/ledger/transaction.rs:254). The actual ledger row is written with a fresh `committed_at: now` at [transaction.rs](C:/dev/ChangeGuard/src/ledger/transaction.rs:294). Since the signature payload includes `committed_at`, hook-driven entries will not verify against their stored fields.

- `Medium` The “skip” path still records intent instead of skipping it, which looks semantically wrong and will pollute provenance/audit data unless this is explicitly intended. Pressing `s` in the TUI sets `"Skipped intent entry"` and returns success at [intent_tui.rs](C:/dev/ChangeGuard/src/ui/intent_tui.rs:361), and `execute_hook_commit_msg` then unconditionally calls `silently_record_ledger` at [hook_commit_msg.rs](C:/dev/ChangeGuard/src/commands/hook_commit_msg.rs:262). The user-facing message says “Intent entry skipped,” but a pending ledger transaction is still created.

- `Medium` Trailer preservation is too permissive and can misclassify ordinary trailing body lines as Git trailers. `extract_trailers` only scans backward for `Token: value` lines at EOF at [hook_commit_msg.rs](C:/dev/ChangeGuard/src/commands/hook_commit_msg.rs:50), but it does not require the blank-line-separated trailer block Git uses. A commit body ending with lines like `Note: ...` or `Context: ...` can be rewritten as trailers even when they are just prose.

- `Low` The new remediation tests are incomplete in the exact areas this track was supposed to harden. `test_non_interactive_bypasses_tui` is only a comment at [tests/hook_commit_msg.rs](C:/dev/ChangeGuard/tests/hook_commit_msg.rs:112), and `test_sign_returns_error_on_missing_key_dir` is also a placeholder at [tests/ledger_crypto.rs](C:/dev/ChangeGuard/tests/ledger_crypto.rs:55). I also did not find coverage for the stale-sidecar/next-commit promotion case or for `require_signing = true` blocking the hook path.

**Open Questions / Assumptions**

- The intended semantics of `s: Skip` are unclear. If “skip” is meant to bypass intent capture, the current implementation is wrong; if it is meant to record a synthetic trivial entry, the UI copy and test names should say that explicitly.
- The `m33` migration tests use a simplified ad hoc `ledger_entries` schema at [m33_migration.rs](C:/dev/ChangeGuard/tests/m33_migration.rs:11), so they do not prove compatibility with the real pre-M33 schema.

`changeguard` commands were blocked in this environment, and I did not run cargo verification in the read-only sandbox, so this review is based on diff inspection and static reasoning only.
