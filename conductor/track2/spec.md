# Technical Specification: Track 2 - Doctor and Platform Detection

## 1. Overview
This specification details the implementation for Phase 3 of the Changeguard CLI, focusing on platform detection and the `doctor` command. This tracks environment characteristics to ensure safe, predictable cross-platform operations and explicitly surfaces mismatches (e.g., Windows vs. WSL environments).

## 2. Functional Requirements
* **Environment Detection:** Identify the runtime OS (Windows, Linux) and specifically detect WSL instances.
* **Shell Detection:** Identify the preferred or active shell (e.g., PowerShell, bash, zsh) for accurate command execution contexts later.
* **Executable Discovery:** Verify the presence of mandatory tools (`git` and `gemini` / `gemini-cli`) in the system `PATH`.
* **Path Classification:** Classify repository path semantics to determine if a workspace is native or mounted (e.g., recognizing `/mnt/c/` in WSL vs Windows-native NTFS).
* **Diagnostics Output:** Implement the `doctor` command to deterministically aggregate and display these checks.
* **Mismatch Warnings:** Detect and surface likely mixed-environment hazards (e.g., repository on a Windows drive accessed from WSL, or Gemini installed only in WSL while working in Windows).

## 3. Architecture & Module Boundaries
* **`src/platform/`**: Dedicated strictly to platform adaptation and environment-specific detection. Contains no application business logic.
  * `mod.rs`: Public API for the platform subsystem.
  * `detect.rs`: OS and WSL detection logic.
  * `shell.rs`: Active shell detection.
  * `paths.rs`: Path classification and normalization.
  * `env.rs`: System PATH queries and executable discovery.
* **`src/commands/doctor.rs`**: The CLI handler that orchestrates calls to the `platform/` module and constructs user-facing diagnostics.

## 4. Engineering Principles & Idiomatic Rust
* **Error Handling:** 
  * All public functions in `platform/` must be fallible and return `Result<T, E>`.
  * Use `thiserror` for typed domain errors within libraries if required, or `anyhow::Result<T>` for internal composition.
  * The command handler (`doctor.rs`) must return `miette::Result<()>` to supply actionable diagnostic messages.
  * Zero use of `unwrap()` or `expect()` in production code. Use `match` branching to handle expected fallible paths.
* **Determinism:**
  * Output generated by the `doctor` command must be predictably ordered (e.g., sorted checks).
  * Test outputs and assertions must remain stable regardless of the local test environment by using isolated mock paths or environments.
