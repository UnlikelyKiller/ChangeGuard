**Findings**

- High: the new `project_symbols.metadata` path is not actually used by the persisted entrypoint classifier, so J7’s Rust false-positive filtering is largely defeated after indexing. [src/index/orchestrator.rs](C:/dev/ChangeGuard/src/index/orchestrator.rs:1129) reloads symbols with only `id, file_id, symbol_name, symbol_kind, is_public` and reconstructs `Symbol { metadata: BTreeMap::new() }` at [src/index/orchestrator.rs](C:/dev/ChangeGuard/src/index/orchestrator.rs:1224). But the new logic in [src/index/entrypoint.rs](C:/dev/ChangeGuard/src/index/entrypoint.rs:67) depends on `metadata["abi"]`, `metadata["macro"]`, `metadata["cfg"]`, and `metadata["reexport"]`. The parser now populates those keys in [src/index/languages/rust.rs](C:/dev/ChangeGuard/src/index/languages/rust.rs:76), and they are written to SQLite in [src/index/orchestrator.rs](C:/dev/ChangeGuard/src/index/orchestrator.rs:1696), but the classification pass drops them. Net effect: you pay the storage/write cost of the new column, but `extern "C"`, proc-macros, feature gates, and re-exports can still be misclassified in the main indexed flow. I also didn’t find a regression test covering persisted metadata -> classify -> dead-code behavior.

- Medium: BM25 snippets now inject ANSI control sequences into machine-readable outputs. [src/search/tantivy_engine.rs](C:/dev/ChangeGuard/src/search/tantivy_engine.rs:136) turns Tantivy HTML snippets into terminal escapes with `.replace("<b>", "\x1b[1m")`, and [src/commands/search.rs](C:/dev/ChangeGuard/src/commands/search.rs:191) forwards that snippet into bridge/JSON output. That contaminates NDJSON/JSON consumers with terminal formatting bytes. I also didn’t find targeted tests for snippet output shape or JSON cleanliness.

- Medium: the KG “timeout” is not a real timeout for long Cozo calls. [src/impact/enrichment/kg_provider.rs](C:/dev/ChangeGuard/src/impact/enrichment/kg_provider.rs:26) only checks elapsed time before/after phases, but the expensive work is still synchronous `cozo.run_script(...)` at [src/impact/enrichment/kg_provider.rs](C:/dev/ChangeGuard/src/impact/enrichment/kg_provider.rs:72), [src/impact/enrichment/kg_provider.rs](C:/dev/ChangeGuard/src/impact/enrichment/kg_provider.rs:95), and [src/impact/enrichment/kg_provider.rs](C:/dev/ChangeGuard/src/impact/enrichment/kg_provider.rs:136). If one query stalls for 90s, a `kg_timeout_secs = 60` config will not degrade gracefully; it will just block for 90s and warn later. That misses the milestone goal.

- Medium: the metadata column adds write/storage amplification without normalization, and today much of that cost is wasted. Every symbol upsert rewrites `metadata=excluded.metadata` in [src/index/orchestrator.rs](C:/dev/ChangeGuard/src/index/orchestrator.rs:1704), storing raw attribute strings like full `#[cfg(...)]` text from [src/index/languages/rust.rs](C:/dev/ChangeGuard/src/index/languages/rust.rs:107). Because the main classifier currently ignores persisted metadata, this is mostly extra DB size and churn on `project_symbols` for little runtime benefit. At minimum, this needs a round-trip test and probably a tighter representation if the column grows beyond a few booleans/short enums.

- Medium: SCIP ingestion does not populate the new metadata, so Rust projects indexed through SCIP will still miss these annotations even after the classifier bug is fixed. [src/scip/symbol_table.rs](C:/dev/ChangeGuard/src/scip/symbol_table.rs:68) hardcodes `metadata: None`. That makes the milestone behavior backend-dependent, which is an architectural inconsistency.
