1. High: `ai-brains` CLI fallback can hang the caller indefinitely, so Milestone B is not actually fail-open when AI-Brains is present but unresponsive. [client.rs](C:/dev/ChangeGuard/src/bridge/client.rs:24) uses `cmd.output()` with no timeout or kill path; Track B4 only handles missing executable / non-zero exit, not the hanging process case. If B6 wires this into `ask`, one stuck vault call will stall the main CLI path.

2. High: timed-out IPC connects leak blocked threads. [ipc.rs](C:/dev/ChangeGuard/src/bridge/ipc.rs:16) spawns a thread to `open()` the named pipe and returns after `recv_timeout`, but the worker thread is never cancelled or joined if `open()` stays blocked. [notify.rs](C:/dev/ChangeGuard/src/bridge/notify.rs:27) does this from a fire-and-forget thread after every `verify`, so repeated runs against a hung pipe can accumulate orphaned blocked threads.

3. High: the B6 `ask` integration is effectively missing. Track B6 requires `ask` to call `bridge::client::query_unified()` and fail open to local-only context when external recall fails, but `query_unified` is only defined in [client.rs](C:/dev/ChangeGuard/src/bridge/client.rs:7) and never used by `ask`. Instead, `ask` only renders whatever was previously persisted into `ImpactPacket.ai_insights` at [ask.rs](C:/dev/ChangeGuard/src/commands/ask.rs:180). That means no live dual-retrieval, and external context can be stale or absent unless a separate import step already mutated `latest-impact.json`.

4. Medium: the preferred IPC path is not safe for reads yet. [ipc.rs](C:/dev/ChangeGuard/src/bridge/ipc.rs:51) performs a blocking `read_line()` with no timeout, and the file comment already notes this can block. If IPC is ever used for actual query/response traffic, a connected but silent AI-Brains server can still hang ChangeGuard even though connect timeout exists.

5. Medium: the v0.2 `BridgeRecord` contract is not enforced on input. [model.rs](C:/dev/ChangeGuard/src/bridge/model.rs:29) injects `version` only during serialization, but [model.rs](C:/dev/ChangeGuard/src/bridge/model.rs:48) deserializes straight into the enum and never checks or warns on version mismatch, despite Track B1 requiring a warning on schema-version drift. This makes “strict contract” optimistic rather than real.

6. Medium: export does not match the specified source of truth for ledger deltas and can emit incorrect counts. Track B2 says to read `.changeguard/state/ledger.cozo`, but [export.rs](C:/dev/ChangeGuard/src/bridge/export.rs:10) opens `ledger.db` and [export.rs](C:/dev/ChangeGuard/src/bridge/export.rs:42) hardcodes `files_changed: 1`. That bakes a lossy approximation into the external contract.

7. Medium: the bridge/verify boundary is bidirectionally coupled. [notify.rs](C:/dev/ChangeGuard/src/bridge/notify.rs:3) imports `verify::results::VerificationResult`, while [verify.rs](C:/dev/ChangeGuard/src/commands/verify.rs:413) calls back into `bridge::notify`. Rust will compile this, but architecturally it means bridge transport now depends on verify internals instead of a bridge-owned DTO, which increases the chance of circular churn as both areas evolve.
